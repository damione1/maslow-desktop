### State machine Maslow — matrice firmware (source : `Calibration::requestStateChange`)
Tous les handlers de commande utilisent `anyState` ; le **vrai** gating est dans `requestStateChange(newState)`. Actions utilisateur → état cible → états source autorisés :
- **Retract** → RETRACTING → **tout état**
- **Extend** → EXTENDING → RETRACTED(2), EXTENDEDOUT(4)
- **Take Slack** → TAKING_SLACK → EXTENDEDOUT(4), READY_TO_CUT(7)
- **Calibrate** → CALIBRATION_IN_PROGRESS → EXTENDEDOUT(4), READY_TO_CUT(7), COMPUTING(9)
- **Comply** → RELEASE_TENSION → UNKNOWN(0), EXTENDEDOUT(4), READY_TO_CUT(7), COMPUTING(9)
- **Stop / E-Stop** → tout état
- Transitions internes (non déclenchables par l'utilisateur) : RETRACTED←RETRACTING, EXTENDEDOUT←(EXTENDING/TAKING_SLACK/RELEASE_TENSION/COMPUTING/IN_PROGRESS), COMPUTING←IN_PROGRESS, READY_TO_CUT←(IN_PROGRESS/COMPUTING/TAKING_SLACK).

> **State machine robuste (FAIT)** : les changements d'état remontés par le firmware sont capricieux (rapports bruités/sauts). `policy_for` filtre les actions opérationnelles aux **états stables** (busy = 1/3/5/6/8/9), MAIS **Retract/Stop/E-Stop restent toujours autorisés** (calque fidèle du firmware : `requestStateChange` accepte RETRACTING depuis n'importe quel état, et `$STOP`/`$ESTOP` sont inconditionnels). C'est la **récupération d'un état bloqué** : `$STOP` arrête les moteurs et remet FluidNC en Idle sans réinitialiser la FSM Maslow → un Stop en EXTENDING(3) gèle l'état ; Retract est la seule sortie. Durcissement livré : (1) `valid_transition(from,to)` valide chaque rapport entrant contre le graphe firmware ; (2) `StateTracker` lisse via debounce 400 ms (straggler ignoré / discord accepté+loggé) ; (3) `action_policy` réconcilie FluidNC + Maslow + job en une policy unique, avec Retract découplé du gate `stable` ; (4) burst-poll `$GSTATE` post-action (~3 s @ 250 ms) pour que l'UI converge en < 1 s au lieu de 1,5 s. NB : ce n'est PAS une FSM qu'on pilote (le firmware possède l'état) — c'est une **fonction état→policy** + un validateur de transitions ; une lib FSM classique n'apporte rien ici (voir note lib dans le journal).

## Phase 7 — Solver Levenberg-Marquardt client-side (EN COURS)
Objectif : répliquer côté desktop le calcul d'ancrages que le firmware fait on-device, pour (1) **vérifier/visualiser** une calibration, (2) **rejouer un « what-if »** en excluant un waypoint suspect **sans re-mesurer** (gain UX majeur — re-mesurer = minutes de course courroie), (3) écrire les ancrages retenus. Décision produit verrouillée : **solveur interactif complet**, **solver d'abord**, derrière une **interface extensible** (`AnchorSolver`) pour brancher d'autres algos plus tard (robuste, autre paramétrage…).

### Découverte firmware (source : `firmware/FluidNC/src/Maslow/Calibration.cpp`)
- Le firmware **émet les mesures brutes** juste avant chaque recompute : `[MSG:INFO: CLBM:[{bl:..,br:..,tr:..,tl:..},…]]` (`logClbmMeasurements`). C'est l'**entrée** du solver client.
- Il émet aussi ses résultats : `Find Anchors fit: rms=.. max=.. perAnchor=[..] converged=1` et `Find Anchors recompute complete: tl=(..,..) tr=(..,..) brX=.. SSR=.. points=N`. Servent d'**oracle** de validation.
- Algo = **LM partitionné par complément de Schur**. 5 params d'ancrage réduits `[tlX, tlY, trX, trY, brX]` (BL **fixé à l'origine**, brY=0) + 2 params position sled par mesure. Init position sled par Gauss-Newton (20 it.). Modèle direct 2D : `dTL=‖s−tl‖, dTR=‖s−tr‖, dBL=‖s‖, dBR=‖s−(brX,0)‖`. Constantes : `λ₀=0.001`, `×10`/`×0.1`, `MAX_ITER=100`, `MAX_REJECT=20`, conv `1e-4`, **10 retries** perturbés (±25/±50 mm, tl/tr symétriques en X), gates **RMS<5 mm** & **maxRes<15 mm** & converged.

## Phase 8.1 — Fix critique : lecture config cassée en WS (HTTP `/command` renvoie vide) — LIVRÉ
**Symptôme** (retour utilisateur + capture) : cliquer « Read » sur le panneau FluidNC (et **aussi** Maslow config) ne remplit rien ; pourtant le YAML `$CD` défile dans la console. **Cause racine confirmée firmware** (`WebServer.cpp:721-728`) : pour toute commande non-`[ESP…]` (donc `$CD`, `$/…`, `$I`), le handler HTTP fait `WSChannels::runGCode(getPageid(), cmd)` et renvoie un **corps vide** (`Content-Length: 0`, vérifié au curl sur 192.168.0.106) — la sortie est routée vers le **canal WS** (`WSChannel::push`), jamais dans la réponse HTTP. Donc les 3 lectures HTTP (`read_maslow_config`/`_anchors`/`read_full_config`) étaient cassées **dès qu'un WS est connecté** (toujours, en pratique).
- **Fix : lecture via capture du flux WS.** Nouveau `WsCommand::DumpConfig` + commande `request_config_dump` : envoie `$CD` sur le WS, accumule les lignes YAML dans `SocketCtx.capture` jusqu'au `ok` terminal, puis `finalize_config_dump` → `flatten_config` → emit `config-dump` (Vec<ConfigEntry>) **+ dérive** `maslow-config`/`maslow-anchors` (rendu synthétique `path=value` re-parsé par `parse_maslow_config`/`parse_anchors`). **Un seul dump remplit les 3 stores.** Durcissements : polls `$MINFO/$GSTATE` suspendus pendant la capture (sinon leurs `ok` la tronqueraient), `ok` ignoré tant que le buffer est vide (anti-race avec un ack de poll résiduel), bruit `[MSG]`/`MINFO:`/control jeté, status reports laissés passer (DRO vivant), timeout 10 s.
- **Front** : `refreshFullConfig`/`refreshConfig`/`refreshAnchors` = alias de `requestConfigDump()` ; stores remplis par les listeners `config-dump`/`maslow-config`/`maslow-anchors`. `read_full_config` HTTP (mort-né) retiré ; `read_maslow_config`/`_anchors` HTTP restent enregistrés mais ne sont plus appelés.
